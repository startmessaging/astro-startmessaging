# Agent Guide — astro-startmessaging

Rules for AI agents working in this Astro project.
**Read this before touching any file.**

---

## Architecture in one diagram

```
global.css (@theme tokens)
    ↓
src/components/ui/          ← primitives  (no content, no Sanity, no i18n logic)
    ↓
src/components/sections/    ← page blocks (consume primitives + r() for i18n)
    ↓
src/content/pages/*.json    ← page manifests (sections[], seo, locales[])
    ↓
src/pages/[...slug].astro   ← routes that read manifests + dispatch via ComponentRenderer
```

---

## Before writing any code

1. **Check `componentMap.ts`** — does a section type already exist?
2. **Check `src/components/ui/`** — can you compose from existing primitives?
3. **Check `src/content/pages/`** — is this a content change, not a code change?
4. **Check `DESIGN_SYSTEM.md`** — every component is inventoried there.

### Component priority (highest first)

| Priority | Action |
|---|---|
| 1 | Change content in a page JSON — no code needed |
| 2 | Add a `variant` prop to an existing section component |
| 3 | Compose existing primitives in an existing section |
| 4 | Create a new section component using primitives |
| 5 | Create a new UI primitive |

## Design system rules (never break these)

- **All colors, sizes, radii, shadows come from `global.css` `@theme` tokens.**
- Use **named Tailwind utilities** generated by `@theme` — never `text-[var(--xxx)]` arbitrary values.
  - Font sizes: `text-display-xl`, `text-body-md`, `text-caption`, `text-button-sm` …
  - Colors: `text-ink`, `text-muted`, `bg-canvas`, `bg-surface-soft`, `border-hairline` …
  - Shadows: `shadow-card`  |  Radius: `rounded-sm`, `rounded-md`, `rounded-full`
- **Never hardcode hex colors or pixel values** in component class lists.
- To change a color/size/radius site-wide: edit `global.css` only.

## i18n rules

| Need | Use |
|---|---|
| Localized string from a page JSON prop | `r(prop, lang)` from `@/i18n/r` |
| Global UI string (nav, footer, CTA) | `t(key, lang)` from `@/i18n/t` |
| Nav / footer link arrays per locale | `NAV_ITEMS[locale]`, `FOOTER_COLUMNS[locale]` from `@/i18n/nav` |
| Generate a localized URL | `getRelativeLocaleUrl(lang, path)` from `astro:i18n` |

- Default locale is **`en`** — no URL prefix.
- Other locales get a prefix: `/pt-br/`, `/hi/` …
- Locale keys in JSON and code are **always lowercase**: `pt-br` not `pt-BR`.
- The `hreflang` attribute uses BCP 47 casing: `pt-BR`, `en-US` (defined in `HREFLANG` map).
- Missing translation → build warning (not error). Fallback is default locale.

## Content collections

### Pages (`src/content/pages/**/*.json`)
```jsonc
{
  "locales": ["en", "pt-br"],   // omit = all locales
  "seo": {
    "title":       { "en": "…", "pt-br": "…" },
    "description": { "en": "…", "pt-br": "…" },
    "ogImage":     "/og/page.png"
  },
  "sections": [
    { "type": "hero", "props": { … } },
    { "type": "comingSoon", "locales": ["en"], "props": { … } }
  ]
}
```
- `type` must match a key in `componentMap.ts`.
- `locales` at section level restricts which locales render that section.

### Blog (`src/content/blog/{lang}/*.mdx`)
- English posts: `en/slug.mdx`
- Translated posts: `pt-br/slug.mdx` (same filename = same slug)
- English-only post: only `en/slug.mdx` exists → no Portuguese route generated.

## Routes

| URL pattern | File | Layout |
|---|---|---|
| `/` | `pages/index.astro` | `MarketingLayout` |
| `/{slug}` | `pages/[...slug].astro` | `MarketingLayout` |
| `/{lang}/{slug}` | `pages/[lang]/[...slug].astro` | `MarketingLayout` |
| `/blog/{slug}` | `pages/blog/[slug].astro` | `BlogLayout` |
| `/{lang}/blog/{slug}` | `pages/[lang]/blog/[slug].astro` | `BlogLayout` |

Folder structure of `src/content/pages/` mirrors the URL:
- `pages/pricing.json` → `/pricing`
- `pages/whatsapp/index.json` → `/whatsapp`
- `pages/whatsapp/bulk-messaging.json` → `/whatsapp/bulk-messaging`

## UI primitives (`src/components/ui/`)

| Component | Props | Notes |
|---|---|---|
| `Container` | `as?`, `class?` | `max-w-7xl` centred wrapper |
| `Section` | `variant?` (`default`\|`soft`\|`strong`\|`ink`), `id?` | Outer `<section>` + bg |
| `Heading` | `as` (`h1`–`h4`), `size?`, `class?` | Semantic heading, `text-ink` default |
| `Text` | `as?`, `size?`, `tone?`, `class?` | `<p>` / `<span>` / `<dt>` / `<dd>` … |
| `Button` | `variant?`, `size?`, `href?`, `disabled?` | `<a>` when href, `<button>` otherwise |
| `Badge` | `tone?` (`primary`\|`muted`\|`success`) | Pill badge |

## Section components (`src/components/sections/`)

| Type key | Component | Notes |
|---|---|---|
| `hero` | `HeroSection.astro` | Badge + heading + subtitle + dual CTAs + optional image |
| `statsBar` | `StatsBar.astro` | `<dl>` stat grid on soft background |
| `featuresGrid` | `FeaturesGrid.astro` | Icon + title + description cards |
| `ctaBlock` | `CtaBlock.astro` | Full-width CTA on ink background |
| `comingSoon` | `ComingSoon.astro` | Placeholder for unbuilt pages |

All section components must:
- Use `Section` + `Container` as the outer shell.
- Use `Heading`, `Text`, `Button`, `Badge` for all text and actions.
- Never contain raw Tailwind colour/size tokens — delegate to primitives.

## Adding a new section

1. Add `type` + `props` schema to `src/content.config.ts` (or rely on loose `z.record(z.unknown())`)
2. Create `src/components/sections/MySection.astro` using primitives
3. Register in `src/components/componentMap.ts`
4. Add a row to `DESIGN_SYSTEM.md` → Section components table
5. Add to any relevant `src/content/pages/*.json`

---

## Anti-patterns

- `text-[var(--color-ink)]` — use `text-ink` instead
- `rounded-[var(--radius-md)]` — use `rounded-md` instead
- Hardcoded `#222222` or `1rem` in class lists
- Raw `<h2>` / `<p>` inside section components when `Heading`/`Text` exist
- Seeding page JSON arrays without `_key` (Sanity rule — N/A here, but keep JSON consistent)
- Creating a one-off component for a single page — use a `variant` prop or `comingSoon` until built
